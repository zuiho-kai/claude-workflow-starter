# 下结论的纪律

## 什么时候用

- 准备说明根因、影响、修复状态或“没有问题”时。
- 静态阅读、日志、测试和用户观察给出不同信号时。
- 结果看起来正常，但缺少已知正确基线时。

## 1. 明确区分推理和实测

没有实际运行时写“从源码推断”；只有跑过并能指出命令、输入和证据时才写“实测”。不要用确定语气让读者误以为已经验证。

```text
推理：从当前分支的调用链看，错误可能发生在请求转换阶段。
实测：运行 <命令> 后，日志 <位置> 显示字段在转换前存在、转换后丢失。
```

## 2. 不符合预期的现象是根因入口

当代码、文档或维护者给出一个应满足的 invariant，而实测不满足时：

1. 记录预期和观察值；
2. 查清差异从哪一步开始；
3. 在因果链闭合前，不把差异解释成“另一种实现”或“可以接受”。

## 3. 声明“无影响”必须写完整链路

准备说 harmless、无需修复或不影响结果时，必须说明：

- 当前值怎样沿调用链传播；
- 在哪里被丢弃、覆盖或隔离；
- 是否影响延迟、算力、内存、状态、日志和其他消费者；
- 用什么证据证明这些副作用不存在或可接受。

只看最终输出相同，不足以证明过程没有问题。

## 4. Exception 是向上追踪的起点

遇到 crash、类型错误或 contract violation 时，先问错误值为什么进入当前路径。修复错误值或错误分支的来源，不要仅绕开抛错位置。

## 5. 用户纠正会使旧结论失效

用户指出事实、目标或判断错误后：

1. 停止继续包装旧结论；
2. 回到当前 source、diff、日志或运行结果；
3. 明确写出旧结论为什么不成立；
4. 重新验证后再下结论。

用户判断是高优先级线索，不替代证据；但重复为旧判断辩护也不是证据。

## 6. 没有 reference 只能报告有限结果

没有改动前、上游或其他已知正确基线时，只能报告当前运行没有 crash、满足哪些明确断言。不能仅凭单边观察声称“无退化”“质量正常”或“完全等价”。

验收输出前主动列出最可能的三个失败模式，并逐项寻找反证，而不是只列成功特征。

## 最小结论格式

```text
结论：<当前能够证明的最强结论>
证据：<source / diff / 命令 / 日志 / artifact>
边界：<尚未验证或不能推出什么>
下一步：<如需扩大结论，还缺哪项验证>
```
