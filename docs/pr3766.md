这次 PR #3766 的目标很明确：让 HunyuanImage3 的 DiT 阶段能接入 diffusion step execution，也就是多个请求可以按 denoise step 分组执行。默认配置不变，只有在 DiT 配了 **step\_execution: true** 且 **max\_num\_seqs > 1** 时才启用。

我先讲整体链路。请求进入后，每个 request 自己保留 scheduler、latents、timesteps、model kwargs 和 CFG 信息。真正执行 DiT 时，不是把整个请求一次性混在一起，而是在 **denoise\_step** 里取当前 active states，把处在同一个 step 的请求 merge 成一个 batch，然后 forward 一步，再 split 回每个 request。这个入口在 **pipeline\_hunyuan\_image3.py** 的 **denoise\_step**，merge 的核心在 **\_merge\_step\_model\_inputs**。

这里第一个关键点是 request-local state。以前单请求执行时，很多东西可以挂在 pipeline 或 layer 上；但 step batching 后，active batch composition 会变，比如 A 先进入 DiT，B 后进入，下一轮可能 A+B 一起跑。所以 PR 里把 scheduler、latents、prompt KV、AR KV、model kwargs 都放到每个 **DiffusionRequestState.extra** 里，避免不同请求互相污染。

第二个关键点是 prompt KV cache。HunyuanImage3 第一轮会 cache prompt KV，后续 denoise step 复用。这里不能只保存 key/value，还必须保存每一行真实的 **image\_kv\_cache\_lens**。否则 active batch 顺序变化时，layer 上残留的 lens 可能来自上一轮 batch，位置校验和 KV 拼接都会错。现在 **\_capture\_prompt\_kv\_cache** 会按 state 和 CFG branch 保存 key/value 以及 lens；**\_restore\_prompt\_kv\_cache** 会按当前 active states 的顺序重新拼回 layer。可以看 **pipeline\_hunyuan\_image3.py** 570 到 647 行。

第三个关键点是不同 prompt 长度。为了让不同 prompt co-batch，merge 时会右 padding **input\_ids**、**position\_ids**、**attention\_mask**、**custom\_pos\_emb** 这些序列字段。**attention\_mask** 是 4D bool mask，短 prompt padding 出来的行和列必须是 false。这个逻辑在 **\_right\_pad\_step\_tensors** 和 **\_merge\_step\_model\_inputs**。

这里引出最容易出错的 attention backend 问题。HunyuanImage3 有 **full\_attn\_spans**，FlashAttention piecewise path 只适合 batch 内 mask 可以被同一套 span 表达的情况。如果 spans 不同，或者 spans 看起来相同但某一行因为 padding 多了 false 区域，FlashAttention piecewise 不能表达这个 per-row mask，而且它会优先看 **full\_attn\_spans**，忽略 4D **attention\_mask**。所以现在 merge 阶段会生成 **requires\_sdpa\_for\_full\_attn\_spans** flag：只要某一行因为 padding 导致 mask 不能被 piecewise FA 精确表达，就标记需要 SDPA。这个 flag 从 merge 贯穿到 **prepare\_inputs\_for\_generation**、**\_update\_model\_kwargs\_for\_generation**、**forward\_call**、**HunyuanImage3Model.forward**，最后到 **AttentionMetadata.extra**。

然后 attention 层的逻辑很小：在 **layer.py** 的 **\_run\_local\_attention**，如果 backend 是 FlashAttention，并且有 **full\_attn\_spans**，只要 spans 非同构，或者 metadata 标了 **requires\_sdpa\_for\_full\_attn\_spans**，就 fallback 到 SDPA。这样 homogeneous spans 的 fast path 还能保留，但不再把 padding token 当成有效 token。

最后讲测试。测试覆盖四类风险。第一，**test\_hunyuan\_image3\_step\_execution\_pads\_different\_prompt\_lengths** 验证不同 prompt 长度 merge 后 mask、position、full spans 和 SDPA flag 都正确。第二，**test\_hunyuan\_image3\_full\_attn\_sdpa\_flag\_survives\_prepare\_and\_update** 验证这个 flag 不会在 prepare 或 generation update 中丢失。第三，**test\_hunyuan\_image3\_prompt\_kv\_cache\_restore\_uses\_current\_state\_order\_and\_lens** 验证 active batch 顺序变化时 KV 和 lens 都按当前 state 顺序恢复。第四，attention fallback 测试验证 homogeneous spans 但显式标记 row-specific padding 时会走 SDPA。

总结一下，这个 PR 的核心不是简单把 tensor cat 起来，而是保证 step batching 后 request-local 状态、prompt KV lens、attention mask 语义都不串。性能方面，PR body 也没有夸大：官方 IT2I 两请求 e2e 是 modest speedup，DiT-only 有更清楚的 isolated speedup；正确性和调度支持是这次主目标。
