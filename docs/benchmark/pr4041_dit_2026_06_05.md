## 性能差异定位：#3766 vs #4041

#3766 的设计在 `full_attn_spans` 非同构或 row-specific padding 时会回退 SDPA：pipeline merge 产出 `requires_sdpa_for_full_attn_spans` flag，attention layer 看到 `FLASH_ATTN + full_attn_spans + flag/nonhomogeneous` 后直接走 `sdpa_fallback`。#4041 则移除了这条 Hunyuan transformer SDPA escape flag，改成在 `flash_attn.py` 里把 `attn_mask` 传给 `piecewise_attn`，由 piecewise FA 自己做 heterogeneous spans / padding-only mask compaction。

2026-06-05 做了最小对照：保持 #4041 head、同一 grouped YAML、同一模型、同一 `CUDA_VISIBLE_DEVICES=5,7`、同一 2 prompts / 50 steps / FP8 / no CFG，只设置 `DIFFUSION_ATTENTION_BACKEND=TORCH_SDPA`。

产物目录：`/data/wzr/pr4041_grouped_sdpa_20260605_172338/`

| Mode | Attention | Elapsed | Throughput |
|---|---|---:|---:|
| baseline request-mode | default | `15.5205s` | `0.12886 req/s` |
| #4041 grouped | `FLASH_ATTN` piecewise with mask compaction | `202.2987s` | `0.00989 req/s` |
| #4041 grouped | forced `TORCH_SDPA` | `15.9735s` | `0.12521 req/s` |

定位结论：主要退化来自 #4041 的 masked piecewise FlashAttention path，不是 step execution / grouped scheduling 本身。强制 SDPA 后，grouped path 从 `202s` 回到 `16s`，基本接近 request-mode baseline。

静态证据链：

- #3766：`vllm_omni/diffusion/attention/layer.py` 遇到非同构 `full_attn_spans` 或 `requires_sdpa_for_full_attn_spans` 时回退 SDPA。
- #3766：`pipeline_hunyuan_image3.py` 在 merge 后发现 row padding 会设置 `requires_sdpa_for_full_attn_spans`。
- #4041：`flash_attn.py` 对 `full_attn_spans` 直接调用 `piecewise_attn(..., attn_mask=attention_mask)`。
- #4041：`piecewise_attn.py` 的 masked path 每 row 做 `_piecewise_mask`、`torch.nonzero`、`.tolist()`、`index_select`、span compaction，再调用 FA。这些操作被放大到每层每 denoise step。

5-step 轻量 profile 进一步确认大头在 masked `piecewise_attn` 顶层路径：

- 产物：`/data/wzr/pr4041_piecewise_profile_20260605_174838/profile_summary.json`
- workload：#4041 grouped default FA，2 prompts，5 measured steps，FP8，no CFG。
- measured elapsed：`20.7378s`。
- 跨 worker 聚合 `piecewise_attn` 顶层：`512 calls / 51.7708s`。
- 已单独包住的 helper 很小：`_piecewise_mask 0.343s`、`_padding_keep_masks 0.263s`、`_compact_spans 0.236s`、`_piecewise_attn_grouped 0.496s`。

解释：顶层 `piecewise_attn` 里的未包裹区包括 `torch.nonzero`、`.tolist()`、`kept_positions.index(...)`、per-row `index_select` / `index_copy_`。其中 `torch.nonzero` 和 `.tolist()` 在 CUDA tensor 上会引入 host-device synchronization / data transfer，是当前 FA 额外耗时的重点嫌疑。

因此，“#3766 虽然架构有问题但有收益”的关键原因很可能是它没有在这个 workload 上走 #4041 的逐 row masked piecewise FA；它用 SDPA 保住 correctness，也绕开了 per-row Python/GPU bookkeeping。#4041 为了避免 SDPA fallback，把 fallback 成本换成了更重的 mask compaction 成本，在 batch=2/no-CFG 场景下反而极慢。

更具体地说，#4041 的 `piecewise_attn(..., attn_mask=attention_mask)` 在 masked 分支没有走 `_piecewise_attn_grouped` 的跨 row grouping，而是 `for row in full_attn_spans` 后每行单独做 `nonzero`、`.tolist()`、`index_select`、单 row `_piecewise_attn_grouped`、`index_copy_`。所以 batch 虽然在 scheduler 层合进来了，attention FA 这块又被拆成 per-row 串行小 kernel / 同步路径；SDPA 则保持一次 batched attention 调用。

2026-06-08 又做了一个 b2 / 5-step 计数诊断，只统计 `piecewise_attn` 的 masked 分支形态，产物在 `/data/wzr/pr4041_piecewise_mask_stats_20260608/`：

- worker 聚合：`512` 次 `piecewise_attn`，全部是 `masked_calls`
- 行数：`896` rows
- `baseline_equal_rows=576`，`no_padding_rows=576`
- `multi_row_calls=384`，其中 `all_rows_same_signature_calls=192`
- `call_unique_signature_hist={'1': 192, '2': 320}`

这说明实现不只是“为了 correctness 处理复杂 mask”，而是没有 fast path：大量 row 的 mask 已经等价于 piecewise baseline、没有 padding，或者整次 call 内 row signature 完全相同，本可以继续走 grouped attention，却仍被统一送进 per-row compaction。另一个设计问题是每层每 step 都从 dense `attention_mask` 反推 `query_keep/key_keep`，而 padding / prefix lens 这些信息在 pipeline merge 时已经知道，不该在 attention kernel wrapper 里重复用 CUDA tensor `nonzero` / `.tolist()` 发现。

## 退化归因口径

这轮慢因不要写成“环境波动”。共享 GPU 会影响绝对时间，但 baseline 和 grouped 使用同一物理 GPU 与同一 FP8 配置，不能解释 `13x` 差距。

更可信的归因是：PR4041 Hunyuan grouped path 进入 step execution 后，把原本 pipeline 内部一次跑完的 50-step denoise loop 拆成 50 次 scheduler/executor step；其中最主要的实测退化点是 masked piecewise FlashAttention。每一步都增加：

- `InputBatch` merge/split；
- request-local `model_kwargs`、attention mask、`full_attn_spans` padding/还原；
- prompt KV cache capture/restore，其中 first step 后有逐层 `detach().clone()`；
- masked piecewise attention 的逐 row `nonzero` / `tolist` / `index_select` bookkeeping。

在 `batch=2`、无 CFG、FP8、2 prompt 的小并发场景下，这些 per-step bookkeeping 没有被更大的 DiT batch 摊薄，反而压过了 grouped forward 的潜在收益。512 sweep 进一步说明：即使 batch 放到 8，默认 FA 仍然慢，因为 masked FA 分支把 attention 按 row 串行化；强制 SDPA 后同一 grouped path 能恢复 `1.77x-3.16x` 收益，问题重点不在 step execution 这个架构入口本身，而在 #4041 选择的 masked piecewise FA 实现。

## 后续 profiling 切口

下一轮要分离“piecewise FA 内部哪一步慢”与“仍残留多少 step bookkeeping”，先跑两个小 profile：

1. 在 #4041 grouped + default FA 下对 `piecewise_attn` 内部计时：`_piecewise_mask`、`_padding_keep_masks`、`nonzero/tolist`、`index_select`、`_piecewise_attn_grouped`。
2. 跑 `step_execution=true, max_num_seqs=1` + SDPA / FA 对照，确认单请求 step path 的残余 overhead。
